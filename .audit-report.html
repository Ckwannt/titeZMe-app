Step 4 — Launch

Creates barberProfiles/{uid}, schedule shard, services, then sets isOnboarded: true, isLive: false, approvalStatus: 'pending', and generates a unique barberCode: TZB-XXXXXX.

Shop owner — not a separate role

Shop owner is not a user type. It's a derived flag ownsShop: true on a barber user. There is no dedicated onboarding flow. The shop is created later via app/dashboard/barber/create-shop/page.tsx.

Redirect-to-onboarding logic  ·  components/RouteGuard.tsx:93–106

If appUser.isOnboarded === false AND route is not /onboarding/*, /, /login, /signup:

  • role === 'client' → redirect to /onboarding/client
  • role === 'barber' → redirect to /onboarding/barber

"Healing" mechanism  ·  lib/auth-context.tsx:102–115

If user has role='barber' AND isOnboarded=false BUT a barberProfiles/{uid} doc exists → auto-flips isOnboarded to true. Recovery path for orphaned half-completed onboardings.

Also: app/api/migrate-barbers-onboarded/route.ts is a backfill endpoint that sets isOnboarded: true on all barberProfiles with isLive: true that are missing the flag.

STEP 3Signup flow

File: app/signup/page.tsx  ·  Two signup paths.

Path A — Email + password

handleSignup, lines 147–211

  1. Collects: role (toggle, default client), firstName, lastName, email, password, confirmPassword.
  2. Calls Firebase createUserWithEmailAndPassword.
  3. Writes users/{uid} with { uid, email, role, firstName, lastName, createdAt, isOnboarded: false, ownsShop: false }.
  4. Sends verification email via sendEmailVerification.
  5. Shows verification screen — user must click email link, then log in.
  6. Does NOT redirect to onboarding directly. Onboarding happens after the user later logs in (RouteGuard catches isOnboarded: false).

Path B — Google sign-in

handleGoogleSignIn, lines 110–145

  1. Pops up Google auth, gets user.
  2. If user doc does NOT exist: creates users/{uid} with role: 'client', isOnboarded: true (skips onboarding), profilePhotoUrl from Google.
  3. If user doc exists: reads role.
  4. Redirects by role: admin → /admin, barber → /dashboard/barber, client → /dashboard/client.
Inconsistency: Google sign-in bypasses onboarding by setting isOnboarded: true immediately. Email signup leaves it false and forces them through onboarding. Google clients never get prompted for phone, city, country, or languages.

STEP 4How user type is determined

The role system

Source of truth: users/{uid}.role — string enum.

Valid runtime values used in code: 'client', 'barber', 'admin'.

Inconsistencies across the role definition:
  • firestore.rules:48 accepts ['client', 'barber', 'shop_owner', 'admin'] in isValidUser() — but 'shop_owner' is never written by the app. CLAUDE.md confirms it should not exist.
  • lib/schemas.ts:6: Zod enum allows only ['client', 'barber'] — no admin, no shop_owner.
  • lib/types.ts:36: TypeScript type allows 'client' | 'barber' | 'admin'.
  • lib/auth-context.tsx:16: AppUser.role only allows 'client' | 'barber'.
  • app/api/cleanup/route.ts:12: queries where('role', '==', 'shop_owner') — dead/broken code.

How "shop owner" is determined

Not a role. It's a boolean flag users/{uid}.ownsShop = true AND barberProfiles/{uid}.ownsShop = true. The user is always still role: 'barber'.

Admin role

Admins are determined via role === 'admin' OR isAdmin === true (see firestore.rules:21–28 and components/AdminGuard.tsx). Admins log in via /admin/login only — app/login/page.tsx:97 actively blocks admin accounts from the public login.
